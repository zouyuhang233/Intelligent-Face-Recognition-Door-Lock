## 一、最终可工作的配置

### 硬件连接
```
MCU(STM32F103ZET6)          摄像头模组(FM225)
  PA2 (USART2_TX)    ────    UART_RX
  PA3 (USART2_RX)    ────    UART_TX
  VCC                ────    5.5V~9V ≥1A
  
MCU                       PC(调试)
  PB10 (USART3_TX)  ────    串口助手 RX (115200)
  PB11 (USART3_RX)  ────    串口助手 TX
```

### 按键功能
| 按键 | MCU引脚 | 功能 | 触发方式 |
|------|---------|------|----------|
| PE3 | PE3 | 录入人脸 | 下拉(GPIO_PIN_RESET) |
| PE2 | PE2 | 验证人脸 | 下拉 |
| PA0 | PA0 | 删除全部用户 | 上拉(GPIO_PIN_SET) |

### 最终录入命令
```c
// main.c PE3 按键处理
FM22x_Enroll(0, 0x1F, 30);
// admin=0(普通用户), face_dir=0x1F(全方向:上下左右正), timeout=30s
```

### 验证命令
```c
// main.c PE2 按键处理
FM22x_Verify(1, 10);
// pd_rightaway=1(立即检测), timeout=10s
```

## 二、核心代码架构

### 文件职责
```
Core/
├── Inc/
│   ├── fm22x.h      # 协议常量、所有函数声明、外部全局变量
│   ├── main.h        # MCU HAL 配置(引脚定义等)
│   └── esp8266.h     # WiFi 模块头(含 cam_rx_byte 声明)
├── Src/
│   ├── fm22x.c       # ★ 核心：协议状态机、命令构建、人脸数据解析
│   ├── main.c        # 主循环：LCD显示、按键处理、命令调度
│   ├── esp8266.c     # HAL_UART_RxCpltCallback(中断回调)、WiFi AT指令
│   ├── usart.c       # USART2(摄像头) USART3(调试) 初始化
│   └── gpio.c        # GPIO 初始化
```

### 数据流（关键架构）
```
摄像头 UART_TX
    │
    ▼
MCU PA3(USART2_RX) ──→ 硬件接收字节
    │
    ▼
USART2_IRQHandler → HAL_UART_IRQHandler → HAL_UART_RxCpltCallback(esp8266.c)
    │
    ├─ USART2 分支:
    │   FM22x_UART_Callback(byte)   ← 逐字节喂给状态机(fm22x.c)
    │       │
    │       ├─ 状态机收集完整帧(EF AA msgid size data checksum)
    │       │
    │       └─ 校验通过 → ProcessReceivedFrame()
    │            │
    │            ├─ MID_REPLY(0x00): 命令回复
    │            │   command_complete = true
    │            │   last_result = 结果码(0=成功,13=超时...)
    │            │   lock_beep_state = 1(验证时)
    │            │
    │            └─ MID_NOTE(0x01): 摄像头主动通知
    │                 ├─ NID_READY(0x00): module_ready = true
    │                 └─ NID_FACE_STATE(0x01): ParseFaceState()
    │                      g_face_updated = true
    │                      g_face_left/top/right/bottom = 人脸框坐标
    │                      g_face_yaw/pitch/roll = 姿态角度
    │                      g_face_state = 人脸状态(0=检测到)
    │                      g_face_count++
    │
    └─ USART3 分支: ESP8266 WiFi 数据(本项目未使用)
    
main() 循环(每100ms):
    │
    ├─ g_face_updated? → LCD显示人脸状态 + USART3串口输出
    │
    ├─ command_complete + enrolling? → 处理录入结果
    ├─ command_complete + verifying?  → 处理验证结果(开锁)
    │
    ├─ 按键检测 → FM22x_Enroll / FM22x_Verify / FM22x_DeleteAll
    │
    └─ 自动关锁(验证成功后3秒)
```

### fm22x.c 全局变量（main.c 通过 extern 引用）
```c
// 状态标志
volatile bool module_ready;       // 摄像头就绪(收到NID_READY)
volatile bool command_complete;   // 命令回复已到达
volatile uint8_t last_result;     // 最后命令的结果码

// 最新人脸状态(中断回调中更新, main循环中读取)
volatile bool g_face_updated;     // 有新的人脸数据
volatile int16_t g_face_state;    // 0=检测到人脸, 1=无脸
volatile int16_t g_face_left, g_face_top, g_face_right, g_face_bottom;
volatile int16_t g_face_yaw;      // 偏航角(0.1°单位, 需/10)
volatile int16_t g_face_pitch, g_face_roll;
volatile uint32_t g_face_count;   // 累计人脸帧数

// 锁/蜂鸣器控制
uint8_t lock_state;               // 1=开锁
uint8_t beep_state;               // 1=蜂鸣
uint8_t lock_beep_state;          // 1=锁+蜂鸣逻辑激活

// USART接收缓冲
uint8_t fm22x_rx_byte;            // HAL_UART_Receive_IT 的接收字节
```

## 三、通信协议核心

### 帧格式
```
EF AA | MsgID | SizeH SizeL | Data[0..N] | Checksum
2B同步 | 1B    | 2B(大端)    | N字节       | 1B(XOR校验)
```

### 校验计算
```c
// 从 MsgID 开始到 Data 末尾，逐字节 XOR
checksum = MsgID ^ SizeH ^ SizeL ^ Data[0] ^ ... ^ Data[N-1]
```

### 关键命令
| 命令 | MsgID | 数据 | 说明 |
|------|-------|------|------|
| VERIFY | 0x12 | pd_rightaway(1B), timeout(1B) | 人脸验证 |
| ENROLL | 0x13 | admin, name[32], face_dir, timeout | **交互式录入(本项目使用)** |
| ENROLL_SINGLE | 0x1D | admin, name[32], face_dir, timeout | 单帧录入(本摄像头不支持) |
| DELETEALL | 0x21 | 无 | 删除全部用户 |
| FACERESET | 0x23 | 无 | 清除录入状态 |
| GETVERSION | 0x30 | 无 | 获取固件版本 |

### NID_FACE_STATE 数据结构（实际固件字节序）
```
★ 注意：官方开发手册写的字段顺序与实际固件不一致！
   以下为实际固件的字节序，已经过数据交叉验证。

data[ 0]    = NID (0x01 = FACE_STATE)
data[ 1.. 2]= left   (int16, LE) ← 人脸框左边距
data[ 3.. 4]= top    (int16, LE) ← 人脸框上边距
data[ 5.. 6]= right  (int16, LE) ← 人脸框右边距
data[ 7.. 8]= bottom (int16, LE) ← 人脸框下边距
data[ 9..10]= yaw    (int16, LE) ← 偏航角(0.1°单位)
data[11..12]= pitch  (int16, LE) ← 俯仰角
data[13..14]= roll   (int16, LE) ← 翻滚角
data[15]    = state  (uint8)     ← 人脸状态(0=检测到,1=无脸)
data[16]    = spare  (uint8)     ← 保留

// 解析代码
int16_t left   = (int16_t)(data[1] | (data[2] << 8));
int16_t top    = (int16_t)(data[3] | (data[4] << 8));
int16_t right  = (int16_t)(data[5] | (data[6] << 8));
int16_t bottom = (int16_t)(data[7] | (data[8] << 8));
int16_t yaw    = (int16_t)(data[9] | (data[10] << 8));
int16_t pitch  = (int16_t)(data[11] | (data[12] << 8));
int16_t roll   = (int16_t)(data[13] | (data[14] << 8));
int16_t state  = data[15];
```

### 结果码
| Code | 宏 | 含义 |
|------|-----|------|
| 0 | MR_SUCCESS | 成功 |
| 1 | MR_REJECTED | 模组拒绝 |
| 4 | MR_FAILED4_CAMERA | 摄像头故障 |
| 9 | MR_FAILED4_MAXUSER | 用户已满(100人) |
| 10 | MR_FAILED4_FACEENROLLED | 人脸已录入 |
| 12 | MR_FAILED4_LIVENESSCHECK | 活体检测失败 |
| 13 | MR_FAILED4_TIMEOUT | 超时(最常见) |

## 四、发现并修复的所有问题

### 问题 1：HAL_UART_AbortReceive_IT 造成接收死窗口
- **文件**：`main.c` PE3 按键处理
- **现象**：录入时中止UART接收→发命令→恢复接收，中间约100ms丢数据
- **修复**：移除 abort/restart，直接发命令（和 Verify 流程一致）

### 问题 2：消息覆盖保护逻辑反效
- **文件**：`fm22x.c` RX_WAIT_CHECKSUM 状态
- **现象**：加了 `!msg_ready || rx_msg_id == MSG_REPLY` 保护，意图防止 REPLY 被 NOTE 覆盖。但摄像头先发占位帧再发真实数据，保护逻辑把占位帧锁死，后续真实数据全被丢弃
- **修复**：回退为始终用最新消息覆盖
- **教训**：单缓冲架构下，最新数据优先于最旧数据

### 问题 3：EnrollItg 命令不被支持
- **文件**：`fm22x.c` FM22x_EnrollItg
- **现象**：发送 MID_ENROLL_ITG (0x26) 后摄像头进入异常状态
- **修复**：弃用此命令

### 问题 4：NID_FACE_STATE 字段顺序——文档与固件不一致 ★★★
- **文件**：`fm22x.c` ParseFaceState
- **现象**：官方开发手册 V1.7 写 `state` 字段在最前面(int16_t)，但实际固件字节序是官方样例代码的顺序(state 在最后, uint8_t)
- **修复**：以实际数据自洽性为准（坐标平滑变化 = 解析正确）
- **教训**：文档≠固件。用多帧变化的数据交叉验证解析是否正确。不要盲目信文档

### 问题 5：EnrollSingle 在此摄像头固件上不工作 ★★★
- **现象**：人脸持续检测到(state=0, 坐标变化)，但 EnrollSingle(0x1D) 始终超时。摄像头内部实际运行的是交互式录入逻辑(state出现250~255的进度标记)
- **解决**：改用 `FM22x_Enroll`(0x13, 交互式录入)，30秒超时，全方向(0x1F)
- **教训**：固件行为可能和文档描述不一致。此摄像头只支持交互式录入

### 问题 6：printf 占用 USART1(烧录口)
- **文件**：`fm22x.c`
- **修复**：删除所有 printf，改为通过全局变量传递数据给 main.c，由 main.c 通过 USART3 输出

### 问题 7：command_complete 被其他命令回复误触发
- **文件**：`main.c`
- **修复**：增加 `enrolling`/`verifying` 标志，只在对应操作进行中才处理 command_complete

### 问题 8：FaceReset 回复被误认为录入成功
- **文件**：`main.c`
- **现象**：FaceReset 发完后 command_complete=true，被 main 循环当成录入成功
- **修复**：FaceReset 后手动 `command_complete = false` 清除

## 五、关键误区总结

| 误区 | 真相 |
|------|------|
| "文档说 state 在最前面" | 实际固件 state 在最后(data[15]) |
| "大端序解析" | 实际数据是小端序 |
| "EnrollSingle 单帧录入更快" | 此摄像头不支持，必须用 Enroll 交互式 |
| "FaceReset 能清除状态帮助录入" | 反而可能干扰，不如直接发 Enroll |
| "printf 调试方便" | 占用了 USART1 烧录口，应该用 USART3 |
| "消息覆盖保护防止丢 REPLY" | 反而锁死占位帧，丢掉真实数据 |
| "yaw 显示 544 说明头偏了 54°" | yaw 单位是 0.1°，544 = 54.4° ÷ 10 ≈ 5.4° |
| "pitch/roll 一直是 0 说明有问题" | 此摄像头固件不计算 pitch/roll，只有 yaw 有值 |

## 六、调试技巧

### 判断摄像头是否在工作
```
1. 上电后 module_ready 应该变 true（收到 NID_READY）
2. 对着人脸时串口持续输出 state=0（检测到人脸）
3. 如果 state=1 且坐标不变 → 摄像头没看到人脸
4. 如果坐标在合理范围(0~640)且帧间变化 → 解析正确
```

### 判断录入是否在进行
```
交互式录入时 state 会出现 250~255 的值：
  - state=254 → 已采集 1 个方向
  - state=253 → 已采集 2 个方向
  - state=252 → 已采集 3 个方向
  - state=251 → 已采集 4 个方向
  - state=250 → 已采集 5 个方向
  全部采集完成后自动返回 SUCCESS
```

### 常见失败原因速查
| 错误码 | 原因 | 解决 |
|--------|------|------|
| 13 TIMEOUT | 人脸不合格/未检测到/方向不够 | 调整姿态、转动头部、确保光线 |
| 10 ENROLLED | 人脸已录入 | 先 PA0 删除再录入 |
| 9 MAXUSER | 100人已满 | 删除部分用户 |
| 4 CAMERA | 摄像头硬件故障 | 检查连接、供电(需≥1A) |

### 供电要求
```
FM225 摄像头模组：5.5V~9.0V，≥1A
红外LED需要充足电流才能正常工作
供电不足 → 人脸检测失败(能通信但无法识别人脸)
```

## 七、交互式录入操作指南

### 按键 PE3 后的操作
```
1. 正对摄像头 30-50cm
2. 缓慢大幅度转头：上→下→左→右
3. 每个方向停住 2 秒(等摄像头采集)
4. LCD 显示 "[X/5] captured!" 说明采集到
5. 5 个方向全部采集完成 → 自动返回 SUCCESS
6. 30 秒内完成即可
```

### 手动转头角度要求(来自开发手册)
```
抬头：正脸向上偏转 5~55 度
低头：正脸向下偏转 5~55 度
左转：正脸向左偏转 8~60 度
右转：正脸向右偏转 8~60 度
```
